# 07_录取数据 数据索引与查询路径

这页不新增任何原始数据，只说明 `07_录取数据` 里目前有什么、适合查什么、查到之后该往哪里回填。

## 一、核心对象位置

- `data/gaokao_2025.db`
  当前 SQLite 主库，实际已承接大量 `2024` 历史录取数据和少量 `2025` 数据样本。
- `sqlite/query.js`
  命令行查询入口，适合按学校 / 专业 / 省份快速抽查。
- `sqlite/import_csv.js`
  批量导入入口。
- `data/checkpoint.json`
  当前抓取进度记录。
- `data/fetch_log.txt`
  当前抓取日志。

## 二、这层数据最适合解决什么问题

### 1. 查历史录取门槛

适合回答：

- 某学校在某省近年大概什么分数段
- 某专业在某省近年大概什么门槛
- 某学校或某专业更像“冲 / 稳 / 保”中的哪一类

### 2. 查省份控制线

适合回答：

- 某省本科线 / 批次线大概在哪
- 某个分数距离控制线有多远

### 3. 查专业门槛高低

适合回答：

- 同省同分段里，某专业是不是偏高门槛
- 相近学校里，哪个专业更容易上

## 三、当前不适合直接回答成确定结论的场景

- “我今年这个分数一定能不能上”
- “你直接告诉我肯定能录取哪所学校”
- “我这个分数今年一定稳不稳”

原因不是不能分析，而是：

- 当前库里仍以 `2024` 历史数据为主
- 部分省份 `school_scores` 还没补满
- 新高考省份的 `major_group` 还不完整
- 用户经常给的是“分数”，而不是“位次”

## 四、最常用查询路径

> 以下命令在项目根目录执行。

### 路径 A：已知学校 + 已知省份

适用问题：

- “陕西考生，想报 XX 大学，历史门槛大概多少？”

推荐查法：

```bash
bun sqlite/query.js --school 学校名 --province 省份
```

优先看：

- `min_score`
- `min_rank`
- `batch`
- `subject`

### 路径 B：已知学校 + 已知专业 + 已知省份

适用问题：

- “山东考生，报 XX 大学软件工程，大概够不够？”

推荐查法：

```bash
bun sqlite/query.js --school 学校名 --major 专业名 --province 省份
```

优先看：

- `major`
- `major_group`
- `min_score`
- `min_rank`

### 路径 C：已知专业 + 已知省份

适用问题：

- “北京考生想学法学，这个专业在本省大概什么门槛？”

推荐查法：

```bash
bun sqlite/query.js --major 专业名 --province 省份
```

适合做：

- 同专业跨学校对比
- 专业门槛高低判断

### 路径 D：只想看本省控制线

适用问题：

- “河南本科线大概多少？”

推荐查法：

```bash
bun sqlite/query.js --lines --province 省份
```

## 五、查完以后该往哪里回填

- 省级线、批次线、分数分布说明：
  回填到 `02_省份数据/省份/省份_录取数据总览.md`
- 学校在某省的历史录取倾向：
  回填到 `03_院校库/省份/学校页`
- 专业在某省的历史门槛：
  回填到 `04_专业库/专业页`
- 适合问答系统直接使用的浓缩内容：
  后续整理进 `11_发布导出`

## 六、使用时必须记住的口径

- 这里首先是“历史数据层”，不是“最终当年结论层”
- 如果主库引用的仍是 `2024` 数据，必须明确告诉用户“仅供参考”
- 如果缺少位次和专业组信息，不要强下录取结论
- 更适合说“倾向判断”和“风险位置”，不适合说“绝对能上”
